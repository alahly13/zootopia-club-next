Backend Runtime And Service Ownership
- The active backend is the Next App Router server surface inside `apps/web/app/api`, not `apps/api`.
- All live route handlers are pinned to the Node runtime because they rely on cookies, Firebase Admin, Firestore, Storage, and multipart uploads.
- Server-only responsibility split
  - `apps/web/app/api/**/route.ts`: request parsing, auth gating, response shaping
  - `apps/web/lib/server/session.ts`: session verification and route/API guards
  - `apps/web/lib/server/repository.ts`: Firestore-backed persistence with a local in-memory fallback when Firebase Admin runtime is unavailable, including active-document normalization plus assessment result lifecycle normalization
  - `apps/web/lib/server/document-runtime.ts`: upload validation, direct-file markdown snapshot orchestration, Storage persistence attempts, and linked-binary reloads for direct-file Assessment generation
  - `apps/web/lib/server/document-markdown.ts`: current local document snapshot helper for upload normalization and text-context fallback continuity
  - `apps/web/lib/server/assessment-linked-document.ts`: Assessment-only linked-document resolution that prefers direct PDF input when the selected provider supports it
  - `apps/web/lib/server/assessment-file-qr.ts`: server-side QR data URL generation for the branded Assessment file surfaces

Auth And Session Architecture
- Session model
  - One secure httpOnly session cookie: `zc_session`
  - Preference cookies: `zc_theme`, `zc_locale`
  - Cookie names are defined in `packages/shared-config/src/env.ts`
- User login flow
  - Public entry point: `/login`
  - Firebase Web SDK Google sign-in happens client-side
  - Client auth persistence is intentionally in-memory only during bootstrap
  - `POST /api/auth/bootstrap` verifies the Google ID token server-side, creates the session cookie, mirrors/upserts the user document, and returns the correct redirect target
  - Client signs out its Firebase Web state after bootstrap so the session cookie remains the authoritative long-lived session
- Admin login flow
  - Public entry point: `/admin/login`
  - Separate UI and separate bootstrap path
  - Admin identifier resolution happens server-side first
  - Real credential step is Firebase Email/Password
  - `POST /api/auth/admin/bootstrap` verifies recent sign-in, provider `password`, allowlisted email, and `admin: true` claim before issuing the same session cookie type
- Route protection
  - `apps/web/proxy.ts` only checks cookie presence and redirects unauthenticated requests optimistically
  - Real authorization stays in server code through:
    - `getSessionSnapshot()`
    - `requireAuthenticatedUser()`
    - `requireCompletedUser()`
    - `requireAdminUser()`
- Session user source of truth
  - `apps/web/lib/server/session.ts` verifies the session cookie through Firebase Admin Auth
  - The decoded token is combined with the Firestore user record through `upsertUserFromAuth()`
  - Active/suspended state comes from the Firestore mirror

  - `profileCompletedAt`

Important Security Assumptions And Trust Boundaries
- Firebase Admin SDK bypasses Firestore Rules and Storage Rules by design
- Browser clients do not read or write Firestore or Storage directly in the current architecture
- `proxy.ts` is never the final authorization layer
- Admin authority requires both allowlisted identity and Firebase custom claim `admin: true`
- Firestore `role` and `status` are mirrors for UX and admin operations, not the sole authority for privileged access
- Profile completion is enforced server-side and cannot be unlocked by UI state alone
- The session cookie is the authoritative long-lived session
- Firebase Web SDK auth state is cleared after session bootstrap
- Server credentials and provider keys must remain outside the client graph
- The shared admin password model is weak and should eventually be replaced with unique credentials plus MFA

Current Behavior And Boundaries
- Anonymous user
  - can access `/login` and `/admin/login`
  - is redirected away from protected routes by proxy and/or server guards
- Authenticated non-admin with incomplete profile
  - can hold a valid session cookie
  - is redirected to `/settings`
  - cannot access `/`, `/assessment`, `/infographic`, or protected user APIs until completion is saved and re-read
- Authenticated non-admin with completed profile
  - can access the full user workspace
  - lands on `/upload` after successful login or profile completion
  - keeps `/` as the protected overview route rather than the primary upload landing
  - can use uploads and assessment generation
  - sees Infographic as a coming-soon UI-locked surface in protected navigation/cards and on `/infographic`
- Authenticated admin
  - must use the separate admin login flow
  - is protected by claims-first server checks
  - is exempt from the non-admin profile-completion gate
  - lands in `/admin`, not `/upload`, after successful admin login
  - retains infographic studio access in the protected user workspace
- Suspended user
  - cannot start or continue a valid session regardless of route intent
